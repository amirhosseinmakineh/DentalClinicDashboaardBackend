# مستند جامع سیستم تخصیص لید به مشاوران

## 1. هدف سیستم

سیستم تخصیص لید مسئول دریافت لیدهای جدید از منبع خارجی، تشخیص لیدهای تکراری، انتخاب مسیر مناسب برای هر لید و تخصیص آن به مشاوران واجد شرایط است. این سیستم دو مسیر اصلی دارد:

1. **تخصیص لحظه‌ای (RealTime)** برای لیدهایی که در ساعت کاری و با وجود مشاور آنلاین آماده دریافت لید وارد می‌شوند.
2. **صف آفلاین (OfflineQueue)** برای لیدهایی که خارج از ظرفیت لحظه‌ای، خارج از ساعت کاری، یا بدون مشاور آنلاین آماده ثبت می‌شوند.

هدف عملیاتی سیستم این است که لیدهای فوری با الزام تماس سه‌دقیقه‌ای به مشاور آنلاین داده شوند و لیدهای باقی‌مانده به شکل دسته‌ای بین مشاوران حاضر و فعال توزیع شوند.

## 2. اجزای اصلی

### 2.1 سرویس کاربردی تخصیص لید

کلاس `LeadAssignmentService` نقطه مرکزی اجرای چرخه تخصیص است و چهار عملیات اصلی را انجام می‌دهد:

| عملیات | مسئولیت |
| --- | --- |
| `AddLeadsAsync` | دریافت لیدها از یکتانت، حذف موارد تکراری بر اساس شماره تماس، تعیین نوع اولیه لید و ذخیره در دیتابیس |
| `AssignPendingOfflineLeadsAsync` | تخصیص لیدهای صف آفلاین به مشاوران حاضر و واجد شرایط |
| `AssignRealTimeLeadsAsync` | تخصیص لیدهای لحظه‌ای جدید به مشاوران آنلاین آماده دریافت لید |
| `ExpireOverdueRealTimeLeadsAsync` | منقضی‌کردن لیدهای لحظه‌ای که گزارش تماس آن‌ها تا پایان مهلت سه‌دقیقه‌ای ثبت نشده است |

### 2.2 سرویس پس‌زمینه

کلاس `LeadAssignmentBackgroundService` هر یک دقیقه یک بار چرخه کامل تخصیص را اجرا می‌کند. ترتیب اجرای چرخه به این صورت است:

1. دریافت و ذخیره لیدهای جدید.
2. تخصیص لیدهای صف آفلاین.
3. تخصیص لیدهای لحظه‌ای.
4. بررسی و انقضای لیدهای لحظه‌ای معوق.

### 2.3 سرویس‌های دامنه

| سرویس/استراتژی | مسئولیت |
| --- | --- |
| `LeadDomainService` | تشخیص ساعت کاری، شب/خارج ساعت کاری، و نوع تخصیص بر اساس زمان و وضعیت مشاور آنلاین |
| `ScoreBasedRoundRobinLeadAssignmentStrategy` | تخصیص لحظه‌ای با مرتب‌سازی مشاوران بر اساس امتیاز و توزیع Round Robin |
| `OfflineLeadAssignmentStrategy` | تخصیص صف آفلاین به مشاوران حاضر، کامل و فعال در بسته‌های ۵تایی |
| `LeadReportDomainService` | تبدیل نتیجه تماس مشاور به وضعیت نهایی/میانی لید |

### 2.4 ریپازیتوری‌ها

| ریپازیتوری | نمونه مسئولیت‌های مهم |
| --- | --- |
| `LeadAssignmentRepository` | واکشی لیدهای آفلاین در انتظار، لیدهای لحظه‌ای بدون مشاور، لیدهای لحظه‌ای منقضی، شماره‌های موجود و لید متعلق به یک مشاور |
| `ConsultantProfileRepository` | واکشی مشاوران واجد شرایط برای صف آفلاین و مشاوران آنلاین آماده دریافت لید لحظه‌ای |

## 3. مدل‌های داده‌ای کلیدی

### 3.1 LeadAssignment

هر لید با مدل `LeadAssignment` نگهداری می‌شود. فیلدهای مهم:

| فیلد | کاربرد |
| --- | --- |
| `UserName` | نام کاربر/مخاطب دریافت‌شده از فرم |
| `PhoneNumber` | شماره تماس؛ معیار اصلی تشخیص تکراری‌بودن لید |
| `LeadAssignmentState` | وضعیت فعلی لید |
| `ConsultantProfileId` | شناسه مشاوری که لید به او تخصیص داده شده است |
| `AssignedAt` | زمان تخصیص لید به مشاور |
| `AssignmentType` | نوع تخصیص: لحظه‌ای یا صف آفلاین |
| `CreatedAt` | زمان ثبت لید در سیستم |
| `CallDeadlineAt` | آخرین زمان مجاز ثبت گزارش تماس برای لید لحظه‌ای |
| `RequiresThreeMinuteCall` | مشخص می‌کند لید مشمول قانون تماس سه‌دقیقه‌ای هست یا نه |
| `ReportDescription` | توضیحات گزارش تماس |
| `ReportSubmittedAt` | زمان ثبت گزارش تماس |
| `ContactedAt` | زمان تماس/ثبت نتیجه تماس |
| `CallResult` | نتیجه تماس مشاور |

### 3.2 ConsultantProfile

پروفایل مشاور شامل وضعیت آمادگی و امتیاز مشاور است. فیلدهای مهم:

| فیلد | کاربرد |
| --- | --- |
| `IsAvailable` | نشان می‌دهد مشاور حضور خود را ثبت کرده است یا نه |
| `IsCompleteProfile` | فقط مشاور با پروفایل کامل وارد فرآیند تخصیص می‌شود |
| `IsOnline` | نشان می‌دهد مشاور آماده دریافت لید لحظه‌ای است |
| `CurrentScore` | امتیاز فعلی مشاور؛ در اولویت تخصیص اثر دارد |
| `WorkStartTime` / `WorkEndTime` | زمان شروع و پایان حضور |
| `LastOnlineAt` / `LastOfflineAt` | آخرین زمان آنلاین/آفلاین شدن |
| `CallAssignments` | لیدهای تخصیص‌یافته به مشاور |

## 4. وضعیت‌ها و انواع لید

### 4.1 نوع تخصیص لید

| مقدار | توضیح |
| --- | --- |
| `RealTime = 1` | لید لحظه‌ای؛ باید سریع به مشاور آنلاین تخصیص داده شود و مشمول قانون سه دقیقه است |
| `OfflineQueue = 2` | لید صف آفلاین؛ در بسته‌های آفلاین به مشاوران حاضر تخصیص داده می‌شود و مهلت سه دقیقه ندارد |

### 4.2 وضعیت لید

| وضعیت | معنی عملیاتی |
| --- | --- |
| `New` | لید لحظه‌ای تازه ثبت شده و هنوز به مشاور تخصیص داده نشده است |
| `Assigned` | لید به مشاور تخصیص داده شده است |
| `Contacted` | گزارش تماس ثبت شده ولی نتیجه آن تبدیل یا رد قطعی نیست؛ مثل تماس برقرار شد، پاسخ نداد یا نیاز به پیگیری دارد |
| `Pending` | لید در صف آفلاین قرار دارد و هنوز به مشاور تخصیص داده نشده است |
| `Converted` | لید به نتیجه مطلوب/تبدیل رسیده است |
| `Expired` | لید لحظه‌ای در بازه سه‌دقیقه‌ای تعیین تکلیف نشده است |
| `Rejected` | لید رد شده یا شماره اشتباه بوده است |

### 4.3 نتیجه تماس

| نتیجه تماس | وضعیت لید بعد از ثبت گزارش |
| --- | --- |
| `Converted` | `Converted` |
| `Rejected` | `Rejected` |
| `WrongNumber` | `Rejected` |

## 5. قوانین ساعت کاری

ساعت کاری سیستم از **09:00** تا قبل از **21:00** است. هر زمان خارج از این بازه، خارج ساعت کاری/شب محسوب می‌شود.

اثر ساعت کاری:

- فقط در ساعت کاری امکان ایجاد ظرفیت لید لحظه‌ای وجود دارد.
- اگر بعد از ثبت گزارش، زمان خارج از ساعت کاری باشد، مشاور به صورت خودکار آفلاین می‌شود.
- اگر لید لحظه‌ای منقضی شود و سیستم خارج از ساعت کاری باشد، مشاور آفلاین باقی می‌ماند.

## 6. چرخه دریافت و ثبت لید جدید

### 6.1 دریافت از منبع خارجی

سیستم از یک URL فرم/گزارش یکتانت HTML دریافت می‌کند، ردیف‌های جدول را می‌خواند و از ستون‌های مربوطه نام و شماره تماس را استخراج می‌کند. سپس مقادیر با `Clean` پاک‌سازی می‌شوند؛ شامل حذف newline، carriage return، tab و decode کردن HTML.

### 6.2 تشخیص لید تکراری

پس از دریافت لیدهای به‌روز، سیستم شماره تماس‌های موجود را از دیتابیس می‌گیرد و فقط لیدهایی را ذخیره می‌کند که شماره تماس آن‌ها قبلاً ثبت نشده باشد. بنابراین معیار یکتایی فعلی لید، `PhoneNumber` است.

### 6.3 تعیین نوع اولیه لید

برای هر لید جدید:

1. اگر سیستم در ساعت کاری باشد، تعداد مشاوران آنلاین آماده دریافت لید لحظه‌ای محاسبه می‌شود.
2. تعداد لیدهای لحظه‌ای بدون مشاور از این ظرفیت کم می‌شود.
3. خروجی، ظرفیت واقعی لید لحظه‌ای است.
4. لیدهای جدید تا سقف این ظرفیت به صورت `RealTime` و وضعیت `New` ذخیره می‌شوند.
5. بقیه لیدها به صورت `OfflineQueue` و وضعیت `Pending` ذخیره می‌شوند.

## 7. قوانین انتخاب مشاور برای لید لحظه‌ای

مشاور فقط وقتی آماده دریافت لید لحظه‌ای است که همه شرایط زیر برقرار باشد:

1. حذف نشده باشد.
2. پروفایل کامل داشته باشد.
3. حضور خود را ثبت کرده باشد (`IsAvailable = true`).
4. آنلاین باشد (`IsOnline = true`).
5. لید آفلاین باز و تعیین‌تکلیف‌نشده نداشته باشد.
6. لید لحظه‌ای تخصیص‌یافته و گزارش‌نشده نداشته باشد.

این قوانین از گرفتن چند لید همزمان توسط یک مشاور و همچنین آنلاین‌شدن مشاور با لید آفلاین باز جلوگیری می‌کند.

## 8. الگوریتم تخصیص لحظه‌ای

### 8.1 ورودی‌ها

- لیدهای `RealTime` با وضعیت `New` و بدون مشاور.
- مشاوران آنلاین آماده دریافت لید.
- تعداد لیدهای واکشی‌شده حداکثر برابر تعداد مشاوران آماده است.

### 8.2 اولویت‌بندی مشاوران

مشاوران بر اساس موارد زیر مرتب می‌شوند:

1. امتیاز فعلی بیشتر (`CurrentScore` نزولی).
2. شناسه کمتر (`Id` صعودی) در صورت برابر بودن امتیاز.

### 8.3 تخصیص Round Robin مبتنی بر امتیاز

لیدها به ترتیب به مشاوران مرتب‌شده تخصیص داده می‌شوند. اگر تعداد لیدها از تعداد مشاوران بیشتر باشد، توزیع به صورت چرخشی انجام می‌شود؛ البته در سرویس فعلی تعداد لیدهای واکشی‌شده برای RealTime برابر تعداد مشاوران است، پس معمولاً هر مشاور آماده در هر چرخه حداکثر یک لید لحظه‌ای می‌گیرد.

### 8.4 اثر تخصیص روی لید و مشاور

بعد از تخصیص لحظه‌ای:

- `ConsultantProfileId` روی لید تنظیم می‌شود.
- `AssignedAt` برابر زمان فعلی می‌شود.
- وضعیت لید `Assigned` می‌شود.
- نوع تخصیص `RealTime` می‌شود.
- `RequiresThreeMinuteCall = true` می‌شود.
- `CallDeadlineAt = AssignedAt + 3 minutes` می‌شود.
- مشاورانی که لید گرفته‌اند، بلافاصله آفلاین می‌شوند و `LastOfflineAt` آن‌ها برابر زمان تخصیص می‌شود.

آفلاین‌کردن خودکار مشاور بعد از گرفتن لید لحظه‌ای باعث می‌شود تا زمانی که گزارش همان لید ثبت نشده، مشاور لید لحظه‌ای دیگری نگیرد.

## 9. قانون تماس سه‌دقیقه‌ای برای لید لحظه‌ای

هر لید لحظه‌ای بعد از تخصیص، مهلت سه‌دقیقه‌ای دارد. اگر تا قبل از `CallDeadlineAt` گزارش ثبت نشود:

1. لید با وضعیت `Expired` علامت‌گذاری می‌شود.
2. برای مشاور یک لاگ امتیاز منفی از نوع `CallNotCompleted` با مقدار `-10` ثبت می‌شود.
3. `CurrentScore` مشاور ۱۰ امتیاز کم می‌شود.
4. وضعیت آنلاین/آفلاین مشاور بر اساس شرایط زیر تعیین می‌شود:
   - اگر مشاور لید آفلاین باز داشته باشد یا سیستم خارج از ساعت کاری باشد، مشاور آفلاین می‌شود.
   - در غیر این صورت، مشاور دوباره آنلاین می‌شود.

## 10. الگوریتم تخصیص صف آفلاین

### 10.1 شرایط مشاور آفلاین

برای دریافت لید از صف آفلاین، مشاور باید:

1. حذف نشده باشد.
2. پروفایل کامل داشته باشد.
3. حضور خود را ثبت کرده باشد (`IsAvailable = true`).

آنلاین‌بودن برای تخصیص صف آفلاین شرط نیست.

### 10.2 انتخاب لیدهای صف آفلاین

لیدهای واجد شرایط صف آفلاین باید:

1. نوع `OfflineQueue` داشته باشند.
2. وضعیت `Pending` داشته باشند.
3. هنوز مشاور نداشته باشند.

این لیدها بر اساس `CreatedAt` و سپس `Id` صعودی مرتب می‌شوند تا لیدهای قدیمی‌تر زودتر تخصیص بگیرند.

### 10.3 اندازه بسته تخصیص

به ازای هر مشاور واجد شرایط، حداکثر **۵ لید** آفلاین تخصیص داده می‌شود. سرویس در هر چرخه حداکثر `تعداد مشاوران × 5` لید Pending واکشی می‌کند.

### 10.4 اثر تخصیص آفلاین روی لید

بعد از تخصیص صف آفلاین:

- `ConsultantProfileId` تنظیم می‌شود.
- `AssignedAt` برابر زمان فعلی می‌شود.
- وضعیت لید `Assigned` می‌شود.
- نوع تخصیص `OfflineQueue` می‌شود.
- `RequiresThreeMinuteCall = false` می‌شود.
- `CallDeadlineAt = null` می‌شود.

## 11. ثبت حضور و آنلاین/آفلاین شدن مشاور

### 11.1 ثبت حضور (`SetAvalableConsultant`)

وقتی مشاور حضور خود را ثبت می‌کند:

- اگر پروفایل کامل و حذف‌نشده باشد، `IsAvailable = true` می‌شود.
- `WorkStartTime` برابر زمان فعلی ثبت می‌شود.

وقتی عدم حضور ثبت می‌شود:

- `IsAvailable = false` می‌شود.
- `IsOnline = false` می‌شود.
- `WorkEndTime` و `LastOfflineAt` برابر زمان فعلی ثبت می‌شوند.

### 11.2 آنلاین شدن (`SetOnlineOfflineConsultant`)

مشاور برای آنلاین‌شدن باید:

1. وجود داشته باشد.
2. حذف نشده باشد.
3. پروفایل کامل داشته باشد.
4. حضور خود را ثبت کرده باشد.
5. لید آفلاین باز و تعیین‌تکلیف‌نشده نداشته باشد.

اگر همه شرایط برقرار باشد، `IsOnline = true` و `LastOnlineAt` برابر زمان فعلی می‌شود.

### 11.3 آفلاین شدن دستی

در حالت آفلاین شدن دستی:

- `IsOnline = false` می‌شود.
- `LastOfflineAt` برابر زمان فعلی می‌شود.

## 12. ثبت گزارش تماس

مشاور فقط برای لیدی می‌تواند گزارش ثبت کند که با `LeadAssignmentId` و `ConsultantProfileId` متعلق به خودش پیدا شود.

### 12.1 اعتبارسنجی‌های ثبت گزارش

ثبت گزارش رد می‌شود اگر:

- لید پیدا نشود.
- مشاور پیدا نشود.
- پروفایل مشاور حذف شده باشد.
- پروفایل مشاور کامل نباشد.
- لید قبلاً `Expired` شده باشد.
- برای لید قبلاً گزارش ثبت شده باشد.

### 12.2 اثر ثبت گزارش روی لید

بعد از ثبت گزارش:

- `CallResult` برابر نتیجه ارسالی می‌شود.
- `ReportDescription` ذخیره می‌شود.
- `ReportSubmittedAt` برابر زمان فعلی می‌شود.
- `ContactedAt` برابر زمان فعلی می‌شود.
- `LeadAssignmentState` بر اساس نتیجه تماس تعیین می‌شود.

### 12.3 اثر ثبت گزارش روی مشاور

بعد از ثبت گزارش:

1. اگر مشاور هنوز لید آفلاین باز داشته باشد، آفلاین می‌ماند/می‌شود و پیام تعیین تکلیف لیدهای آفلاین برگردانده می‌شود.
2. اگر خارج از ساعت کاری باشد، مشاور آفلاین می‌شود.
3. در غیر این صورت، مشاور به صورت خودکار آنلاین می‌شود تا بتواند لید لحظه‌ای بعدی را دریافت کند.

2. شماره تماس در دیتابیس وجود ندارد.
3. ظرفیت لحظه‌ای مثبت است.
4. لید با نوع `RealTime` و وضعیت `New` ذخیره می‌شود.
5. در مرحله تخصیص RealTime، لید به مشاور با بالاترین امتیاز تخصیص می‌یابد.
6. مشاور آفلاین می‌شود تا گزارش همان لید را ثبت کند.
7. اگر گزارش تا سه دقیقه ثبت شود، وضعیت بر اساس نتیجه تماس تغییر می‌کند و در صورت نبود مانع، مشاور دوباره آنلاین می‌شود.
8. اگر گزارش ثبت نشود، لید `Expired` می‌شود و مشاور ۱۰ امتیاز منفی می‌گیرد.

### 14.2 لید خارج از ساعت کاری وارد می‌شود

1. لید دریافت و تکراری نبودن آن بررسی می‌شود.
2. چون خارج ساعت کاری است، لید به صورت `OfflineQueue` و وضعیت `Pending` ذخیره می‌شود.
3. در چرخه تخصیص آفلاین، اگر مشاور حاضر واجد شرایط وجود داشته باشد، لید به او تخصیص داده می‌شود.
4. این لید مهلت سه‌دقیقه‌ای ندارد.

### 14.3 مشاور با لید آفلاین باز می‌خواهد آنلاین شود

1. مشاور درخواست آنلاین‌شدن می‌دهد.
2. سیستم وجود لید آفلاین باز را بررسی می‌کند.
3. اگر لید آفلاین تعیین‌تکلیف‌نشده وجود داشته باشد، درخواست آنلاین‌شدن رد می‌شود.
4. مشاور باید ابتدا گزارش لیدهای آفلاین خود را ثبت کند.

### 14.4 مشاور گزارش لید لحظه‌ای را ثبت می‌کند

1. سیستم مالکیت لید توسط مشاور را بررسی می‌کند.
2. اگر لید منقضی یا گزارش‌شده نباشد، گزارش ثبت می‌شود.
3. وضعیت لید از روی نتیجه تماس تعیین می‌شود.
4. اگر مشاور لید آفلاین باز ندارد و سیستم داخل ساعت کاری است، مشاور خودکار آنلاین می‌شود.

## 15. نکات مهم طراحی و محدودیت‌های فعلی

1. معیار تشخیص لید تکراری فقط شماره تماس است.
2. ساعت کاری در کد ثابت است و از تنظیمات خوانده نمی‌شود.
3. اندازه بسته صف آفلاین ثابت و برابر ۵ است.
4. مشاور بعد از دریافت لید لحظه‌ای خودکار آفلاین می‌شود تا فقط یک لید لحظه‌ای باز داشته باشد.
5. لیدهای آفلاین مشمول قانون سه‌دقیقه‌ای نیستند.
6. لیدهای لحظه‌ای فقط وقتی منقضی می‌شوند که وضعیت `Assigned` داشته باشند، گزارش نداشته باشند و `CallDeadlineAt` گذشته باشد.
7. وجود لید آفلاین باز، مانع آنلاین‌شدن و دریافت لید لحظه‌ای جدید است.
8. اولویت تخصیص بر اساس امتیاز فعلی مشاور است؛ در امتیاز برابر، شناسه کوچک‌تر اولویت دارد.

## 16. پیشنهادهای بهبود آینده

1. انتقال ساعت کاری و اندازه بسته آفلاین به تنظیمات قابل تغییر.
2. اضافه‌کردن لاگ ساختاریافته برای هر چرخه تخصیص و هر تغییر وضعیت لید.
3. اضافه‌کردن تست واحد برای استراتژی‌های تخصیص RealTime و OfflineQueue.
4. اضافه‌کردن تست یکپارچه برای چرخه کامل `LeadAssignmentBackgroundService`.
5. جلوگیری از ثبت رکوردهای ناسالم با اعتبارسنجی شماره تماس قبل از ذخیره.
6. افزودن شاخص دیتابیسی روی `PhoneNumber`، `AssignmentType`، `LeadAssignmentState` و `ConsultantProfileId` برای بهبود کارایی.
7. امکان تعریف قوانین پویا برای امتیازدهی، جریمه و اولویت تخصیص.
8. اضافه‌کردن گزارش مدیریتی برای نرخ تبدیل، نرخ انقضا، عملکرد مشاور و زمان متوسط پاسخ‌گویی.
