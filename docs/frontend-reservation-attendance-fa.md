# مستند جامع فرانت: رزرو، تایید حضور، کنترل صف آفلاین و لید لحظه‌ای

## 1. هدف تسک
این تسک برای این طراحی شده که:

1. مشاور هنگام رزرو، اطلاعات تکمیلی بیمار را ثبت کند:
   - شهر بیمار
   - درصد احتمال حضور بیمار در مطب
   - توضیح/پیش‌بینی مشاور درباره اینکه بیمار در روز و ساعت رزرو می‌آید یا نه
2. بعد از رسیدن زمان رزرو، سیستم به مشاور اعلام کند که باید حضور/عدم حضور بیمار را مشخص کند.
3. تایید مشاور نهایی نباشد؛ چون ممکن است مشاور برای گرفتن امتیاز، حضور بیمار را اشتباه ثبت کند.
4. منشی، تایید مشاور را بررسی کند و در صورت تایید یا رد، سیستم به صورت خودکار امتیاز مثبت یا منفی برای مشاور ثبت کند.
5. فرانت فقط زمانی لیدهای لحظه‌ای را به مشاور نشان دهد که قانون الگوریتم رعایت شده باشد:
   - مشاور رزروهای موعددار و تعیین‌تکلیف‌نشده برای حضور بیمار نداشته باشد.
   - تکلیف لیدهای صف آفلاین مشاور مشخص شده باشد.

> طبق درخواست بک‌اند، هیچ migration در این تسک ساخته نشده است. قبل از اجرای واقعی، ستون‌های جدید باید با فرآیند دیتابیسی مورد تایید تیم آماده شوند.

## 2. قانون مهم الگوریتم برای نمایش لید لحظه‌ای
فرانت نباید صرفا هر زمان که لیست لید لحظه‌ای خالی نبود آن را نمایش دهد. بک‌اند هم روی `GetLeads` این قانون را اعمال می‌کند، اما فرانت باید UX درست را پیاده کند.

### 2.1 مشاور چه زمانی حق دیدن لید لحظه‌ای ندارد؟
اگر هر کدام از موارد زیر برقرار باشد، مشاور نباید لید لحظه‌ای ببیند:

1. رزروی وجود دارد که:
   - زمان آن رسیده یا گذشته است.
   - لغو نشده است.
   - وضعیت آن هنوز `PendingConsultantConfirmation` است.
   - یعنی مشاور هنوز حضور/عدم حضور بیمار را نزده است.
2. لید آفلاینی برای مشاور وجود دارد که:
   - نوع آن `OfflineQueue` است.
   - هنوز گزارش تماس برای آن ثبت نشده است (`reportSubmittedAt == null`).
   - وضعیت آن نهایی نشده است؛ یعنی `Expired`، `Rejected` یا `Converted` نیست.

### 2.2 رفتار فرانت در این حالت
وقتی مشاور وارد صفحه لیدهای لحظه‌ای می‌شود:

1. ابتدا `GET /api/Reservation/DueConfirmations?consultantProfileId={id}` را صدا بزنید.
2. اگر خروجی آیتم داشت، بخش لید لحظه‌ای را قفل کنید و کارت‌های «نیازمند تایید حضور» را نشان دهید.
3. بعد از اینکه مشاور همه موارد را با `POST /api/Reservation/ConfirmAttendance` تعیین تکلیف کرد، دوباره وضعیت را refresh کنید.
4. سپس لیست صف آفلاین مشاور را با `GET /api/Consultant/GetLeads?profileId={id}&leadAssignmentType=2` بررسی کنید.
5. تا وقتی صف آفلاین تعیین تکلیف نشده، به جای لیست لید لحظه‌ای پیام بدهید: «ابتدا لیدهای آفلاین خود را تعیین تکلیف کنید.»
6. بعد از تعیین تکلیف حضورها و صف آفلاین، `GET /api/Consultant/GetLeads?profileId={id}&leadAssignmentType=1` را برای نمایش شماره‌های لحظه‌ای صدا بزنید.

## 3. ثبت رزرو توسط مشاور
### Endpoint
`POST /api/Reservation`

### Payload
```json
{
  "leadAssignmentId": 10,
  "consultantProfileId": 3,
  "reservationAt": "2026-06-27T10:30:00",
  "patientCity": "تهران",
  "attendanceProbabilityPercent": 80,
  "attendancePrediction": "بیمار گفت روز و ساعت رزرو شده داخل مطب حاضر می‌شود.",
  "description": "توضیح اختیاری"
}
```

### اعتبارسنجی‌های بک‌اند
- `reservationAt` باید آینده باشد.
- `patientCity` اجباری است.
- `attendanceProbabilityPercent` باید بین `0` تا `100` باشد.
- `attendancePrediction` اجباری است.
- فقط لیدهای تماس موفق قابل رزرو هستند.
- برای یک لید، رزرو فعال تکراری ثبت نمی‌شود.
- ظرفیت هر بازه زمانی برای مشاور کنترل می‌شود.

### Response مهم برای فرانت
در پاسخ، فیلدهای جدید برمی‌گردد:

```json
{
  "id": 25,
  "leadAssignmentId": 10,
  "consultantProfileId": 3,
  "patientUserId": null,
  "requiresPatientProfile": true,
  "reservationAt": "2026-06-27T10:30:00",
  "patientCity": "تهران",
  "attendanceProbabilityPercent": 80,
  "attendancePrediction": "بیمار گفت روز و ساعت رزرو شده داخل مطب حاضر می‌شود.",
  "attendanceConfirmationStatus": 1,
  "patientName": "علی احمدی",
  "patientPhoneNumber": "09120000000"
}
```

## 4. نوتیفیکیشن/لیست تایید حضور برای مشاور
### Endpoint
`GET /api/Reservation/DueConfirmations?consultantProfileId=3`

### کاربرد
این endpoint رزروهایی را می‌دهد که:
- متعلق به مشاور هستند.
- زمانشان رسیده یا گذشته است.
- لغو نشده‌اند.
- هنوز مشاور حضور/عدم حضور بیمار را تعیین نکرده است.

### رفتار پیشنهادی UI
- در هدر داشبورد مشاور badge نشان بدهید.
- در صفحه مشاور یک بخش «تایید حضورهای لازم» بگذارید.
- تا وقتی این لیست خالی نشده، بخش شماره‌های لحظه‌ای را قفل کنید.

## 5. تایید حضور توسط مشاور
### Endpoint
`POST /api/Reservation/ConfirmAttendance`

### Payload برای بیمار حاضر شده
```json
{
  "reservationId": 25,
  "consultantProfileId": 3,
  "patientAttended": true,
  "note": "بیمار در مطب حاضر شد."
}
```

### Payload برای بیمار حاضر نشده
```json
{
  "reservationId": 25,
  "consultantProfileId": 3,
  "patientAttended": false,
  "note": "بیمار در زمان رزرو حاضر نشد."
}
```

### محدودیت‌ها
- مشاور فقط بعد از رسیدن زمان رزرو می‌تواند این عملیات را انجام دهد.
- اگر منشی قبلا بررسی کرده باشد، مشاور دیگر نمی‌تواند وضعیت را تغییر دهد.
- بعد از ثبت مشاور، رکورد منتظر بررسی منشی می‌ماند.

## 6. تسک فرانت برای داشبورد/بخش منشی
فعلا داشبورد منشی نداریم، پس فرانت باید یک بخش جدید اضافه کند؛ مثلا:

- مسیر پیشنهادی: `/secretary/reservation-attendance-reviews`
- عنوان پیشنهادی: «بررسی تایید حضور مشاورها»

### اطلاعاتی که باید در جدول منشی نمایش داده شود
- نام بیمار
- شماره بیمار
- شهر بیمار
- مشاور
- زمان رزرو
- درصد احتمال حضور
- متن پیش‌بینی مشاور
- وضعیت ادعای مشاور: آمده / نیامده
- یادداشت مشاور
- دکمه تایید ادعای مشاور
- دکمه رد ادعای مشاور
- یادداشت منشی

### Endpoint بررسی منشی
`POST /api/Reservation/ReviewAttendance`

### Payload تایید منشی
```json
{
  "reservationId": 25,
  "secretaryUserId": "00000000-0000-0000-0000-000000000000",
  "approved": true,
  "note": "با حضور بیمار در مطب تطبیق داده شد."
}
```

### Payload رد منشی
```json
{
  "reservationId": 25,
  "secretaryUserId": "00000000-0000-0000-0000-000000000000",
  "approved": false,
  "note": "طبق بررسی منشی، تایید مشاور درست نبود."
}
```

### اثر امتیازی
- اگر `approved=true` باشد، سیستم برای مشاور امتیاز مثبت ثبت می‌کند.
- اگر `approved=false` باشد، سیستم برای مشاور امتیاز منفی ثبت می‌کند.
- این امتیاز فقط یک بار اعمال می‌شود.

## 7. وضعیت‌های تایید حضور
فرانت مقدار `attendanceConfirmationStatus` را دریافت می‌کند:

| عدد | نام | معنی | اقدام فرانت |
| --- | --- | --- | --- |
| 1 | PendingConsultantConfirmation | منتظر تایید مشاور | نمایش در لیست DueConfirmations |
| 2 | ConsultantConfirmedPresent | مشاور گفته بیمار آمده | نمایش در صف بررسی منشی |
| 3 | ConsultantConfirmedAbsent | مشاور گفته بیمار نیامده | نمایش در صف بررسی منشی |
| 4 | SecretaryApproved | منشی تایید مشاور را قبول کرده | فقط نمایش وضعیت نهایی |
| 5 | SecretaryRejected | منشی تایید مشاور را رد کرده | فقط نمایش وضعیت نهایی |

---

## 8. لیست شماره لید لحظه‌ای برای تست
برای تست سناریوی شماره‌های لحظه‌ای، فرانت باید یک لیست مستقل اضافه کند.

### Endpoint
`GET /api/Consultant/GetLeads?profileId=3&leadAssignmentType=1&pageNumber=1&pageSize=10`

`leadAssignmentType=1` یعنی `RealTime`.

### رفتار الزامی
- قبل از نمایش این لیست، قوانین بخش 2 را اجرا کنید.
- اگر بک‌اند به خاطر blocker لیست خالی برگرداند، پیام واضح نمایش دهید:
  - «برای مشاهده لید لحظه‌ای ابتدا حضورهای موعددار را تعیین تکلیف کنید.»
  - یا «برای مشاهده لید لحظه‌ای ابتدا صف آفلاین خود را تعیین تکلیف کنید.»
- این لیست برای تست باید شماره، نام، وضعیت و نوع لید را نشان دهد.

### نمونه Response
```json
{
  "items": [
    {
      "id": 101,
      "leadAssignmentState": 2,
      "leadAssignmentType": 1,
      "phoneNumber": "09120000000",
      "userName": "علی احمدی"
    }
  ],
  "totalCount": 1,
  "pageNumber": 1,
  "pageSize": 10,
  "totalPages": 1
}
```

## 9. لیست صف آفلاین برای تعیین تکلیف
### Endpoint
`GET /api/Consultant/GetLeads?profileId=3&leadAssignmentType=2&pageNumber=1&pageSize=10`

`leadAssignmentType=2` یعنی `OfflineQueue`.

### رفتار پیشنهادی
- اگر آیتم آفلاین تعیین‌تکلیف‌نشده وجود دارد، آن‌ها را بالاتر از لید لحظه‌ای نمایش دهید.
- کاربر باید گزارش تماس هر لید آفلاین را ثبت کند.
- بعد از ثبت گزارش تماس و نهایی شدن وضعیت‌ها، صفحه لید لحظه‌ای refresh شود.

---

## 10. ترتیب پیشنهادی پیاده‌سازی فرانت
1. فرم ثبت رزرو را با سه فیلد جدید آپدیت کنید.
2. کامپوننت DueConfirmations را برای مشاور بسازید.
3. دکمه‌های «بیمار آمد» و «بیمار نیامد» را به DueConfirmations وصل کنید.
4. بخش بررسی منشی را اضافه کنید، حتی اگر فعلا داشبورد منشی کامل ندارید.
5. لیست شماره لید لحظه‌ای تستی را اضافه کنید.
6. قبل از نمایش لید لحظه‌ای، قانون حضورهای تعیین‌تکلیف‌نشده و صف آفلاین را اجرا کنید.
