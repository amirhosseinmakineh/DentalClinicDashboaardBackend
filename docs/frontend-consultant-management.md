# مستند فرانت‌اند: مدیریت مشاور و داشبورد لیدهای مشاور

این سند برای پیاده‌سازی فرانت بخش مشاور آماده شده است و APIهای مرتبط با تکمیل پروفایل مشاور، ثبت حضور، آنلاین/آفلاین شدن، مشاهده لیدهای همان پروفایل، نوشتن گزارش لید و APIهای مرتبط با داشبورد لیدها/رزرو را پوشش می‌دهد.

## قواعد عمومی پاسخ API

اکشن‌های ثبت/تغییر وضعیت معمولاً با این ساختار پاسخ می‌دهند:

```json
{
  "isSuccess": true,
  "message": "پیام عملیات",
  "data": {}
}
```

در خطا نیز معمولاً HTTP Status همچنان `200 OK` است و باید از `isSuccess` تصمیم بگیرید:

```json
{
  "isSuccess": false,
  "message": "پیام خطا",
  "data": null
}
```

لیست‌ها با ساختار زیر برمی‌گردند:

```json
{
  "items": [],
  "totalCount": 0,
  "pageNumber": 1,
  "pageSize": 10,
  "totalPages": 0
}
```

## Enumهای مورد نیاز

### LeadAssignmentState

| مقدار | نام | توضیح برای UI |
| --- | --- | --- |
| `1` | `New` | لید جدید / آماده تماس |
| `2` | `Assigned` | تخصیص داده شده |
| `3` | `Contacted` | تماس موفق / تماس گرفته شده |
| `4` | `Pending` | در انتظار تعیین تکلیف |
| `5` | `Converted` | تبدیل شده |
| `6` | `Expired` | منقضی شده |
| `7` | `Rejected` | رد شده |

### LeadAssignmentType

| مقدار | نام | توضیح |
| --- | --- | --- |
| `1` | `RealTime` | لید لحظه‌ای در ساعت کاری و هنگام وجود مشاور آنلاین |
| `2` | `OfflineQueue` | لید صف آفلاین / خارج از جریان لحظه‌ای |

### LeadCallResult

این مقدار هنگام ثبت گزارش تماس ارسال می‌شود:

| مقدار | نام | اثر کلی روی لید |
| --- | --- | --- |
| `1` | `Contacted` | تماس برقرار شد |
| `2` | `Converted` | تبدیل/موفق شد |
| `3` | `Rejected` | رد شد |
| `4` | `NoAnswer` | پاسخ نداد |
| `5` | `WrongNumber` | شماره اشتباه بود |
| `6` | `NeedFollowUp` | نیاز به پیگیری دارد |

### AttendanceStatus

| مقدار | نام | توضیح |
| --- | --- | --- |
| `1` | `Present` | حاضر |
| `2` | `Absent` | غایب |
| `3` | `Leave` | مرخصی |
| `4` | `SickLeave` | مرخصی استعلاجی |
| `5` | `Late` | تأخیر |
| `6` | `Mission` | مأموریت |

# 1. تکمیل پروفایل مشاور

## API تکمیل پروفایل

| مورد | مقدار |
| --- | --- |
| Method | `POST` |
| URL | `/api/Consultant` |
| Content-Type | `application/json` |

### Request Body

```json
{
  "profileId": 0,
  "nationalityCode": "0012345678",
  "address": "تهران، خیابان مثال، پلاک ۱۰",
  "isCompleteProfile": true
}
```

### فیلدها

| فیلد | نوع | اجباری | توضیح |
| --- | --- | --- | --- |
| `profileId` | `number` | در مدل وجود دارد | در Handler فعلی برای انتخاب پروفایل استفاده نمی‌شود |
| `nationalityCode` | `string` | بله | کد ملی مشاور؛ نام فیلد در API همین `nationalityCode` است |
| `address` | `string` | بله | آدرس مشاور |
| `isCompleteProfile` | `boolean` | پیشنهادی | پیشنهاد فرانت: `true` ارسال شود؛ Handler فعلی پروفایل را کامل ذخیره می‌کند |

### نمونه پاسخ موفق

```json
{
  "isSuccess": true,
  "message": "اطلاعات مشاور با موفقیت تکمیل شد",
  "data": 12
}
```

`data` همان `profileId` ساخته‌شده است و باید برای اکشن‌های بعدی مثل حضور، آنلاین/آفلاین شدن، لیدها و رزروها نگهداری شود.

### نکات مهم فرانت

- قبل از ثبت حضور یا آنلاین شدن، پروفایل باید کامل باشد.
- اگر پروفایل کامل نباشد، APIهای حضور، آنلاین/آفلاین و گزارش لید خطای `پروفایل مشاور کامل نیست` می‌دهند.
- در پیاده‌سازی فعلی Handler، مشاور را بر اساس اولین کاربر دارای نقش `Consultant` پیدا می‌کند و از `profileId` ورودی برای پیدا کردن پروفایل استفاده نمی‌کند؛ اگر چند مشاور هم‌زمان وجود داشته باشد، این رفتار باید در بک‌اند اصلاح شود.

# 2. حضور و غیاب مشاور

در این سیستم، «حضور» با `isAvailable` کنترل می‌شود. مشاور فقط وقتی می‌تواند آنلاین شود که ابتدا حضور خود را ثبت کرده باشد.

## شرایط ثبت حضور

برای ثبت حضور موفق:

1. `profileId` باید معتبر باشد.
2. پروفایل حذف نشده باشد.
3. پروفایل کامل شده باشد.
4. مقدار `isAvailable` برابر `true` ارسال شود.

وقتی حضور ثبت می‌شود:

- `IsAvailable = true` می‌شود.
- `WorkStartTime` برابر زمان فعلی ثبت می‌شود.
- بک‌اند بلافاصله تلاش می‌کند لیدهای آفلاین Pending را به مشاوران حاضر تخصیص دهد.

## API ثبت حضور / عدم حضور

| مورد | مقدار |
| --- | --- |
| Method | `POST` |
| URL | `/api/Consultant/SetAvalableConsultant` |
| Content-Type | `application/json` |

> نام مسیر در بک‌اند فعلی `SetAvalableConsultant` است، نه `SetAvailableConsultant`.

### Request Body برای ثبت حضور

```json
{
  "profileId": 12,
  "isAvailable": true
}
```

### پاسخ موفق ثبت حضور

```json
{
  "isSuccess": true,
  "message": "حضور شما ثبت شد"
}
```

### Request Body برای ثبت عدم حضور / پایان حضور

```json
{
  "profileId": 12,
  "isAvailable": false
}
```

### پاسخ موفق ثبت عدم حضور

```json
{
  "isSuccess": true,
  "message": "عدم حضور شما ثبت شد"
}
```

وقتی عدم حضور ثبت می‌شود:

- `IsAvailable = false` می‌شود.
- `IsOnline = false` می‌شود.
- `WorkEndTime` برابر زمان فعلی ثبت می‌شود.
- `LastOfflineAt` برابر زمان فعلی ثبت می‌شود.

### خطاهای مهم حضور

| سناریو | پیام نمونه |
| --- | --- |
| پروفایل پیدا نشود | `مشاوری یافت نشد` |
| پروفایل حذف شده باشد | `پروفایل مشاور حذف شده است` |
| پروفایل کامل نباشد | `پروفایل مشاور کامل نیست` |

# 3. آنلاین و آفلاین شدن مشاور

آنلاین/آفلاین بودن با `isOnline` کنترل می‌شود. این وضعیت برای دریافت لیدهای RealTime اهمیت دارد.

## شرایط آنلاین شدن

برای آنلاین شدن موفق:

1. `profileId` باید معتبر باشد.
2. پروفایل حذف نشده باشد.
3. پروفایل کامل شده باشد.
4. مشاور باید قبلاً حضور خود را ثبت کرده باشد؛ یعنی `IsAvailable = true` باشد.
5. مشاور نباید لید آفلاین تعیین‌تکلیف‌نشده داشته باشد.

اگر مشاور لید آفلاین Pending داشته باشد، اجازه آنلاین شدن ندارد و باید ابتدا گزارش آن لیدها را ثبت کند.

## API آنلاین/آفلاین شدن

| مورد | مقدار |
| --- | --- |
| Method | `POST` |
| URL | `/api/Consultant/SetOnlineOfflineConsultant` |
| Content-Type | `application/json` |

### Request Body برای آنلاین شدن

```json
{
  "profileId": 12,
  "isOnline": true,
  "isOffline": false
}
```

### پاسخ موفق آنلاین شدن

```json
{
  "isSuccess": true,
  "message": "شما آنلاین شدید"
}
```

### Request Body برای آفلاین شدن

```json
{
  "profileId": 12,
  "isOnline": false,
  "isOffline": true
}
```

> نکته: Handler فعلی فقط `isOnline` را مبنای تصمیم قرار می‌دهد و از `isOffline` استفاده نمی‌کند. با این حال برای خوانایی و سازگاری فرانت، هنگام آفلاین شدن `isOffline: true` ارسال شود.

### پاسخ موفق آفلاین شدن

```json
{
  "isSuccess": true,
  "message": "شما آفلاین شدید"
}
```

### خطاهای مهم آنلاین/آفلاین

| سناریو | پیام نمونه |
| --- | --- |
| پروفایل پیدا نشود | `مشاوری یافت نشد` |
| پروفایل حذف شده باشد | `پروفایل مشاور حذف شده است` |
| پروفایل کامل نباشد | `پروفایل مشاور کامل نیست` |
| حضور ثبت نشده باشد | `ابتدا حضور خود را ثبت کنید` |
| لید آفلاین تعیین‌تکلیف‌نشده وجود داشته باشد | `ابتدا لیدهای آفلاین خود را تعیین تکلیف کنید` |

## ساعت کاری و دریافت لید RealTime

ساعت کاری در بک‌اند از `09:00` تا قبل از `21:00` تعریف شده است.

- اگر در ساعت کاری مشاور آنلاین وجود داشته باشد، لیدهای جدید می‌توانند `RealTime = 1` شوند.
- اگر خارج از ساعت کاری باشد یا مشاور آنلاین آماده وجود نداشته باشد، لیدها در `OfflineQueue = 2` قرار می‌گیرند.
- بعد از تخصیص لید RealTime به مشاور، بک‌اند مشاور را موقتاً آفلاین می‌کند تا ابتدا همان لید تعیین تکلیف شود.
- اگر لید RealTime در بازه مقرر منقضی شود، وضعیت لید `Expired` می‌شود و امتیاز منفی سیستمی ثبت می‌شود.

# 4. لیست لیدهای مرتبط با همان پروفایل

این API برای صفحه «لیدهای من» یا لیست لیدهای مشاور استفاده می‌شود.

| مورد | مقدار |
| --- | --- |
| Method | `GET` |
| URL | `/api/Consultant/GetLeads` |
| نیاز به Body | ندارد |

## Query Parameters

| پارامتر | نوع | اجباری | توضیح |
| --- | --- | --- | --- |
| `profileId` | `number` | بله | شناسه پروفایل مشاور |
| `leadAssignmentState` | `number` | خیر | فیلتر وضعیت لید |
| `leadAssignmentType` | `number` | خیر | فیلتر نوع لید |
| `pageNumber` | `number` | خیر | پیش‌فرض `1` |
| `pageSize` | `number` | خیر | پیش‌فرض `10` |

## نمونه درخواست

```http
GET /api/Consultant/GetLeads?profileId=12&pageNumber=1&pageSize=10
```

## نمونه درخواست برای لیدهای آفلاین Pending

```http
GET /api/Consultant/GetLeads?profileId=12&leadAssignmentState=4&leadAssignmentType=2&pageNumber=1&pageSize=10
```

## نمونه پاسخ

```json
{
  "items": [
    {
      "id": 45,
      "userName": "علی رضایی",
      "phoneNumber": "09121234567",
      "leadAssignmentState": 4,
      "leadAssignmentType": 2
    }
  ],
  "totalCount": 1,
  "pageNumber": 1,
  "pageSize": 10,
  "totalPages": 1
}
```

## نکات فیلتر لیدها

- در Handler فعلی، فیلتر وضعیت فقط برای `New = 1`، `Pending = 4`، `Contacted = 3` و `Rejected = 7` اعمال می‌شود.
- در Handler فعلی، فیلتر نوع فقط برای `OfflineQueue = 2` اعمال می‌شود.
- اگر فرانت به فیلتر دقیق `RealTime`، `Assigned`، `Converted` یا `Expired` نیاز دارد، بک‌اند باید تکمیل شود.

# 5. نوشتن گزارش برای لید

بعد از تماس با بیمار/لید، مشاور باید گزارش تماس را ثبت کند. این اکشن وضعیت لید را بر اساس `callResult` تغییر می‌دهد و ممکن است وضعیت آنلاین مشاور را هم به صورت خودکار تغییر دهد.

## API ثبت گزارش لید

| مورد | مقدار |
| --- | --- |
| Method | `POST` |
| URL | `/api/Consultant/SubmitLeadCallReport` |
| Content-Type | `application/json` |

### Request Body

```json
{
  "leadAssignmentId": 45,
  "consultantProfileId": 12,
  "callResult": 1,
  "reportDescription": "با بیمار تماس گرفته شد و برای رزرو اعلام آمادگی کرد."
}
```

### فیلدها

| فیلد | نوع | اجباری | توضیح |
| --- | --- | --- | --- |
| `leadAssignmentId` | `number` | بله | شناسه لید از لیست لیدهای مشاور |
| `consultantProfileId` | `number` | بله | شناسه پروفایل مشاور؛ باید مالک همین لید باشد |
| `callResult` | `number` | بله | یکی از مقادیر `LeadCallResult` |
| `reportDescription` | `string` | خیر | متن گزارش تماس |

### نمونه پاسخ موفق

```json
{
  "isSuccess": true,
  "message": "گزارش ثبت شد و شما به صورت خودکار آنلاین شدید",
  "data": {
    "leadAssignmentId": 45,
    "consultantProfileId": 12,
    "isReportSubmitted": true,
    "reportSubmittedAt": "2026-06-24T10:30:00",
    "leadAssignmentState": 3,
    "callResult": 1,
    "isConsultantOnline": true
  }
}
```

### پیام‌های موفق احتمالی و اثر آن‌ها روی UI

| پیام | معنی برای فرانت |
| --- | --- |
| `گزارش ثبت شد و شما به صورت خودکار آنلاین شدید` | گزارش ثبت شد و مشاور دوباره آنلاین است |
| `گزارش ثبت شد، اما هنوز لید آفلاین تعیین‌تکلیف‌نشده دارید` | گزارش ثبت شد، ولی مشاور آفلاین می‌ماند تا بقیه لیدهای آفلاین را تعیین تکلیف کند |
| `گزارش ثبت شد، اما خارج از ساعت کاری هستید` | گزارش ثبت شد، ولی به دلیل خارج بودن از ساعت کاری، مشاور آفلاین می‌ماند |

### خطاهای مهم ثبت گزارش

| سناریو | پیام نمونه |
| --- | --- |
| لید برای این پروفایل پیدا نشود | `لید یافت نشد` |
| پروفایل پیدا نشود | `مشاوری یافت نشد` |
| پروفایل حذف شده باشد | `پروفایل مشاور حذف شده است` |
| پروفایل کامل نباشد | `پروفایل مشاور کامل نیست` |
| لید منقضی شده باشد | `مهلت ثبت گزارش این لید به پایان رسیده است` |
| گزارش قبلاً ثبت شده باشد | `گزارش این لید قبلا ثبت شده است` |

## منطق پیشنهادی UI بعد از ثبت گزارش

- همیشه `data.isConsultantOnline` را بخوانید و وضعیت آنلاین/آفلاین UI را با آن Sync کنید.
- اگر پیام مربوط به لیدهای آفلاین باقی‌مانده بود، لیست لیدها را با فیلتر `leadAssignmentState=4&leadAssignmentType=2` دوباره بگیرید.
- اگر `callResult` برابر `Contacted = 1` یا `Converted = 2` بود، امکان ثبت رزرو را در UI نمایش دهید.

# 6. APIهای مرتبط با رزرو در داشبورد لیدها

رزرو زمانی استفاده می‌شود که تماس با لید موفق بوده باشد.

## 6.1 ثبت رزرو برای لید

| مورد | مقدار |
| --- | --- |
| Method | `POST` |
| URL | `/api/Reservation` |
| Content-Type | `application/json` |

### شرایط ثبت رزرو

برای ثبت رزرو موفق:

1. `reservationAt` باید در آینده باشد.
2. مشاور باید وجود داشته باشد و حذف نشده باشد.
3. لید باید متعلق به همان مشاور باشد.
4. گزارش لید باید ثبت شده باشد.
5. نتیجه تماس فقط باید `Contacted = 1` یا `Converted = 2` باشد.
6. برای همان لید نباید رزرو فعال قبلی وجود داشته باشد.
7. ظرفیت همان بازه زمانی برای مشاور نباید پر باشد؛ سقف فعلی ۱۰ رزرو برای هر مشاور در یک زمان یکسان است.

### Request Body

```json
{
  "leadAssignmentId": 45,
  "consultantProfileId": 12,
  "reservationAt": "2026-06-25T14:30:00",
  "description": "رزرو اولیه پس از تماس موفق"
}
```

### نمونه پاسخ موفق

```json
{
  "isSuccess": true,
  "message": "رزرو با موفقیت ثبت شد",
  "data": {
    "id": 1001,
    "leadAssignmentId": 45,
    "consultantProfileId": 12,
    "reservationAt": "2026-06-25T14:30:00",
    "patientName": "علی رضایی",
    "patientPhoneNumber": "09121234567"
  }
}
```

### خطاهای مهم رزرو

| سناریو | پیام نمونه |
| --- | --- |
| زمان رزرو گذشته یا اکنون باشد | `زمان رزرو باید در آینده باشد` |
| مشاور پیدا نشود یا حذف شده باشد | `مشاور یافت نشد` |
| لید متعلق به مشاور نباشد | `لید برای این مشاور یافت نشد` |
| گزارش ثبت نشده یا تماس موفق نباشد | `فقط لیدهای تماس موفق قابل رزرو هستند` |
| رزرو فعال قبلی برای لید وجود داشته باشد | `برای این بیمار قبلا رزرو فعال ثبت شده است` |
| ظرفیت بازه زمانی پر باشد | `ظرفیت این بازه زمانی برای مشاور تکمیل است` |

## 6.2 لیست رزروهای مشاور

| مورد | مقدار |
| --- | --- |
| Method | `GET` |
| URL | `/api/Reservation/GetConsultantReservations` |
| نیاز به Body | ندارد |

### Query Parameters

| پارامتر | نوع | اجباری | توضیح |
| --- | --- | --- | --- |

# 8. فلو پیشنهادی فرانت برای داشبورد مشاور

## ورود به داشبورد

1. `profileId` مشاور را از بک‌اند/State کاربر دریافت کنید.
2. اگر پروفایل کامل نیست، صفحه تکمیل پروفایل را نمایش دهید و `POST /api/Consultant` را صدا بزنید.
3. بعد از تکمیل پروفایل، `profileId` برگشتی را ذخیره کنید.

## شروع روز کاری

1. مشاور روی «ثبت حضور» کلیک می‌کند.
2. فرانت `POST /api/Consultant/SetAvalableConsultant` با `isAvailable: true` را صدا می‌زند.
3. بعد از موفقیت، لیست لیدهای مشاور را بگیرد؛ مخصوصاً لیدهای آفلاین Pending.
4. اگر لید آفلاین Pending وجود داشت، اجازه آنلاین شدن ندهید یا پیام بدهید که ابتدا باید لیدها تعیین تکلیف شوند.
5. اگر لید آفلاین Pending وجود نداشت، دکمه آنلاین شدن را فعال کنید.

## آنلاین شدن

1. فرانت `POST /api/Consultant/SetOnlineOfflineConsultant` با `isOnline: true` را صدا می‌زند.
2. اگر موفق بود، وضعیت مشاور در UI آنلاین شود.
3. اگر خطای `ابتدا لیدهای آفلاین خود را تعیین تکلیف کنید` برگشت، لیست لیدهای آفلاین Pending را نمایش دهید.

## کار با لید

1. لیست لیدها را از `GET /api/Consultant/GetLeads` بگیرید.
2. روی هر لید، فرم گزارش را باز کنید.
3. گزارش را با `POST /api/Consultant/SubmitLeadCallReport` ثبت کنید.
4. بعد از ثبت گزارش، وضعیت آنلاین UI را از `data.isConsultantOnline` به‌روزرسانی کنید.
5. اگر نتیجه تماس موفق بود، امکان ثبت رزرو را نمایش دهید.

## پایان روز کاری

1. برای آفلاین شدن موقت، `POST /api/Consultant/SetOnlineOfflineConsultant` با `isOnline: false` را صدا بزنید.
2. برای پایان حضور، `POST /api/Consultant/SetAvalableConsultant` با `isAvailable: false` را صدا بزنید.

## چک‌لیست خطاناپذیری فرانت

- همیشه `profileId` را از `userId` جدا نگه دارید؛ اکثر APIهای مشاور با `profileId` کار می‌کنند.
- قبل از آنلاین شدن، حتماً حضور ثبت شود.
- قبل از آنلاین شدن، لیدهای آفلاین Pending همان مشاور را چک کنید.
- بعد از ثبت گزارش، وضعیت آنلاین را حدس نزنید؛ از `data.isConsultantOnline` استفاده کنید.
- برای ثبت رزرو فقط وقتی دکمه را فعال کنید که `callResult` برابر `1` یا `2` باشد.
- تاریخ‌های ارسالی مثل `reservationAt`، `from` و `to` را با فرمت ISO 8601 ارسال کنید.
- در خطاهای منطقی، به HTTP Status اتکا نکنید و `isSuccess` و `message` را مبنا قرار دهید.

# 9. منقضی کردن لید RealTime بعد از پایان مهلت سه دقیقه‌ای بدون تماس

این API برای زمانی است که تایمر سه دقیقه‌ای لید RealTime در فرانت تمام شده و مشاور هیچ گزارشی برای تماس ثبت نکرده است. فرانت باید در پایان تایمر، این API را صدا بزند تا بک‌اند:

- وضعیت لید را `Expired = 6` کند.
- امتیاز مشاور را `10-` کم کند.
- یک ScoreLog سیستمی با دلیل `LateCall = 8` ثبت کند.
- وضعیت آنلاین مشاور را بر اساس شرایط فعلی Sync کند.

## API پایان مهلت بدون تماس

| مورد | مقدار |
| --- | --- |
| Method | `POST` |
| URL | `/api/Consultant/ExpireLeadNoCall` |
| Content-Type | `application/json` |

## شرایط اجرای موفق

برای موفق شدن این API:

1. لید باید متعلق به همان `consultantProfileId` باشد.
2. پروفایل مشاور باید وجود داشته باشد.
3. پروفایل حذف نشده باشد.
4. پروفایل کامل باشد.
5. برای لید قبلاً گزارش ثبت نشده باشد.
6. لید قبلاً `Expired` نشده باشد؛ این شرط جلوی کسر امتیاز تکراری را می‌گیرد.
7. لید باید از نوع `RealTime = 1` باشد و `RequiresThreeMinuteCall = true` داشته باشد.
8. اگر `callDeadlineAt` در بک‌اند ثبت شده باشد، زمان فعلی باید از آن عبور کرده باشد.

## Request Body

```json
{
  "leadAssignmentId": 45,
  "consultantProfileId": 12
}
```

## نمونه پاسخ موفق

```json
{
  "isSuccess": true,
  "message": "لید منقضی شد و امتیاز مشاور کسر شد",
  "data": {
    "leadAssignmentId": 45,
    "consultantProfileId": 12,
    "leadAssignmentState": 6,
    "deductedScore": -10,
    "currentScore": 90,
    "isConsultantOnline": true
  }
}
```

## خطاهای مهم

| سناریو | پیام نمونه |
| --- | --- |
| لید متعلق به این مشاور پیدا نشود | `لید یافت نشد` |
| پروفایل مشاور پیدا نشود | `مشاوری یافت نشد` |
| پروفایل حذف شده باشد | `پروفایل مشاور حذف شده است` |
| پروفایل کامل نباشد | `پروفایل مشاور کامل نیست` |
| گزارش قبلاً ثبت شده باشد | `برای این لید قبلا گزارش ثبت شده است` |
| لید قبلاً منقضی شده باشد | `این لید قبلا منقضی شده است` |
| لید RealTime سه دقیقه‌ای نباشد | `این لید مشمول مهلت سه دقیقه‌ای نیست` |
| هنوز سه دقیقه تمام نشده باشد | `مهلت سه دقیقه‌ای این لید هنوز تمام نشده است` |

## رفتار وضعیت آنلاین بعد از انقضای لید

بعد از انقضای لید:

- اگر مشاور لید آفلاین Pending داشته باشد، بک‌اند مشاور را آفلاین نگه می‌دارد.
- اگر خارج از ساعت کاری باشد، بک‌اند مشاور را آفلاین نگه می‌دارد.
- اگر داخل ساعت کاری باشد و لید آفلاین Pending وجود نداشته باشد، بک‌اند مشاور را دوباره آنلاین می‌کند.

فرانت باید بعد از پاسخ موفق، وضعیت لید و وضعیت آنلاین مشاور را از `data.leadAssignmentState` و `data.isConsultantOnline` به‌روزرسانی کند.

## فلو پیشنهادی فرانت برای تایمر سه دقیقه‌ای

1. وقتی لید RealTime به مشاور نمایش داده شد، تایمر سه دقیقه‌ای UI را شروع کنید.
2. اگر مشاور قبل از پایان تایمر گزارش ثبت کرد، دیگر `ExpireLeadNoCall` را صدا نزنید.
3. اگر تایمر تمام شد و هنوز گزارش ثبت نشده بود، `POST /api/Consultant/ExpireLeadNoCall` را با `leadAssignmentId` و `consultantProfileId` صدا بزنید.
4. اگر پاسخ موفق بود، لید را در UI با وضعیت `Expired = 6` نمایش دهید و امتیاز/وضعیت آنلاین مشاور را Sync کنید.
5. اگر پاسخ خطا بود، پیام `message` را نمایش دهید و لیست لیدها را دوباره از `GET /api/Consultant/GetLeads` دریافت کنید.
